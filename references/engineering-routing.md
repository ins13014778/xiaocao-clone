# Engineering Routing

## 目标

根据任务类型快速选择正确的工程路径，不要所有任务都用同一套机械流程。

## 新功能开发

默认顺序：

1. 先备份
2. 先读完整项目
3. 写项目理解文档
4. 沟通方案、接口、数据库、返回值、页面效果
5. 写规格文档
6. 一次实现一个功能
7. 测试这个功能
8. 写交付文档
9. 用户确认后进入下一个功能

## Bug 排查

默认顺序：

1. 先看报错和日志
2. 看关键调用链
3. 查环境问题
4. 查参数和配置
5. 查数据库状态
6. 不够再查 Google、Bing 或合适的 MCP
7. 连修三次失败就升级到官方文档和一手资料

## 项目重构

默认顺序：

1. 先备份
2. 判断重构是否真的为当前功能服务
3. 做必要的小范围重构
4. 不借功能之名做无关大重构
5. 回归验证

## UI / 前端任务

优先考虑：

- Figma MCP
- Pencil
- Playwright
- chrome-bridge

流程：

1. 先看设计和页面预期
2. 先确认页面效果
3. 再进入实现
4. 网站场景优先用 Playwright MCP 做验证

## 嵌入式任务

优先考虑：

1. 官方文档
2. 芯片 / 模块手册
3. 通信协议和关键参数
4. 最小可运行验证

## 部署运维任务

优先排查：

- 环境变量
- 环境问题
- 端口和反向代理
- 日志
- 启动命令
- 守护进程
- 资源占用
- 安全性
